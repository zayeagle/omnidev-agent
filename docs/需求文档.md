# TUI 终端编码 Agent 完整需求文档

> **项目名称**：omnidev-agent（TUI 终端自主代码智能体）
> **对标产品**：Claude Code、Cursor Agent、Codex CLI、Gemini CLI
> **文档用途**：开发约束、功能定义、交付验收标准
> **文档状态**：v1.4 / 功能规格含 v2.1 Skills+MCP

---

## 1. 项目概述

### 1.1 项目目标

从零实现最小可用 **TUI（Terminal User Interface）终端编码 Agent**，程序运行于终端，提供图形化结构化交互界面。用户输入自然语言开发指令后，Agent 读取当前仓库上下文、自主规划任务、调度工具完成代码编写、文件操作、命令执行，循环推理直至任务完成或失败终止。

### 1.2 核心考察能力

评审重点验证以下自研工程实现能力：

1. **Agent Loop** — 自主推理循环调度
2. **工具调用系统** — 标准化设计
3. **权限控制** — 文件与 Shell 操作分级管控
4. **LLM 抽象层** — 多 Provider 接入
5. **会话管理** — 上下文存储与持久化
6. **TUI 界面** — 终端可视化交互
7. **配置管理** — 分层配置 + 严格网关兼容模式
8. **容错工程** — 流式输出、超时、重试等异常处理

## 2. 功能需求规范

### 2.1 TUI 终端图形交互界面（硬性必实现）

程序必须以终端图形界面形式运行，界面需完整展示全链路信息：

1. 用户自然语言输入指令
2. LLM 模型流式输出及完整回复内容
3. 工具调用全流程：工具名称、入参、执行状态
4. 工具执行返回结果与错误堆栈信息
5. 高危操作权限确认弹窗及用户拒绝提示
6. 当前任务阶段、运行状态与执行进度

界面交互能力要求：结构化输入输出、实时状态刷新、弹窗式人机确认交互。

### 2.2 Agent 自主推理闭环流程

标准循环链路：

```
LLM 模型决策 → 工具调用执行 → 结果回传模型 → 模型持续推理
```

1. 接收用户指令，自动读取本地代码仓库上下文，解析任务意图
2. 自主拆分多步骤执行方案，按需自动调用内置工具集
3. 将工具执行结果与用户授权记录注入会话上下文，交由模型判断后续动作
4. 循环终止三条件：任务全部完成、执行失败终止、生成最终总结回复

### 2.3 内置工具系统

所有工具提供标准化入参、出参、错误返回结构，供 LLM 自主调度。

#### 自动执行类工具（无需用户确认）

| 工具 | 说明 |
|------|------|
| 目录浏览 | 遍历项目文件树结构 |
| 文件读取 | 读取文本文件完整内容 |
| 文件匹配 | 按名称或后缀模糊匹配文件 |
| 代码检索 | 关键词、正则全文搜索代码片段 |
| 文件写入 | 新建、覆盖文件内容 |
| 文件编辑 | 增量修改、局部替换代码片段 |

> 文件读写操作视为常规开发行为，不列为危险操作，自动执行无需弹窗确认。

#### 高危操作类工具（必须用户确认后执行）

| 工具 | 说明 |
|------|------|
| Shell 命令执行 | 运行构建、测试、脚本、系统级命令等终端命令 |
| 文件删除 | 删除文件或目录 |

### 2.4 分级权限安全控制（强制规则）

| 操作类型 | 权限策略 |
|----------|----------|
| 常规操作（读文件、写文件、编辑文件、浏览目录、代码检索） | 无确认弹窗，自动执行 |
| 高危操作（运行 Shell、执行系统级命令、删除文件/目录） | TUI 弹窗确认，等待用户授权 |

用户拒绝授权分支逻辑：终止本次工具调用，拒绝记录写入会话上下文，模型基于拒绝记录调整后续执行方案。

### 2.5 LLM 对接与协议兼容要求

1. 支持接入 OpenAI、Anthropic、DeepSeek 等主流大模型标准协议（OpenAI 协议兼容的厂商无需额外开发，仅需配置 BaseURL）
2. **兼容严格 OpenAI 网关模式**（`compat_mode: strict`）：合并 system 消息、省略 temperature、限制 max_tokens、支持 SSE 响应解析
3. 配套容错工程能力：
   - 工具调用请求结构化解析
   - 模型输出流式实时打印
   - 请求超时拦截与自动重试机制
   - 模型返回异常兜底处理逻辑

### 2.6 会话上下文与持久化存储

1. **内存会话**：完整存储用户指令、模型回复、工具调用记录、权限拒绝记录，支撑多轮连续推理
2. **持久化落地（双目录）**：
   - `./.ai_history/sessions/` — **omnidev-agent 运行时** TUI/headless 会话快照（JSON/MD，程序自动 Save/Export）
   - `./.ai_history/logs/` — **开发协作**期间外部 AI 助手（Cursor、Codex 等）的全量对话与工具记录（见 `AGENTS.md`）

### 2.7 分层配置管理模块

支持全局与项目两级配置，可配置项包含：

- LLM API 密钥、模型名称、请求超时、并发参数
- 工具权限默认控制策略
- TUI 界面样式、日志输出等级
- 严格网关兼容参数（`compat_mode`）

### 2.8 运行验收能力标准

交付后的 Agent 可在真实本地环境完整执行复杂开发任务。标准执行链路示例：

```
读取项目文件 → 检索现有代码逻辑 → 新建或修改代码文件 → 执行测试 Shell 命令 → 根据报错迭代修改代码 → 直至任务完成
```

### 2.9 Skills 与 MCP 扩展 (v2.1)

#### Skills（SKILL.md）

1. 支持 Cursor 兼容的 `SKILL.md` 工作流目录结构：`<skills_dir>/<name>/SKILL.md`
2. 默认扫描路径：`~/.omnidev-agent/skills/`、项目内 `.omnidev-agent/skills/`
3. 提供工具 `list_skills`、`load_skill`；TUI 命令 `/skills`、`/skill <name>`
4. 加载后的技能正文以 `[SKILL: name]` 注入会话上下文，供后续推理遵循

#### MCP（Model Context Protocol）

1. 通过配置 `mcp_servers` 启动 stdio MCP 子进程，自动 `initialize` + `tools/list`
2. 远端工具注册为 `mcp_<server>__<tool>`，走统一工具调度与权限模型
3. 每服务器可配置 `tool_level`（safe / dangerous）、`env`、`cwd`、`disabled`
4. MCP 工具输出受 PARTIAL + spool 预算约束，与内置工具一致
5. 单服务器连接失败不影响其他服务器与主 Agent 启动

## 3. 开发实现约束（红线规范）

### 3.1 允许使用第三方库

可直接引入成熟第三方库实现**非核心底层能力**，无限制：

- LLM 模型协议接入 SDK（OpenAI、Anthropic 官方或社区库）
- TUI 终端渲染、终端交互 UI 库
- HTTP 请求客户端、配置解析、日志打印、单元测试工具

### 3.2 严禁依赖第三方 Agent 框架（核心逻辑必须自研）

禁止使用 LangChain、LlamaIndex、AutoGen 等任意 Agent SDK 或框架实现以下模块，全部逻辑须自主编码设计：

1. **Agent Loop** — 多轮推理循环调度
2. **工具系统** — 定义、解析、调度、结果处理
3. **权限控制** — 文件与命令权限校验逻辑
4. **会话管理** — 上下文存储与历史记忆管理

> 评审重点：自主设计的接口、调度流程、任务决策逻辑，不允许封装第三方框架完成核心 Agent 流程。

### 3.3 开源项目复用规范

1. 允许查阅开源同类项目获取功能与架构设计灵感
2. 禁止以任何开源项目为基底修改后提交（包括 Fork、Clone 二次改动、复制代码重命名项目）
3. 领域通用分层结构（工具层、上下文层等）不视为抄袭；评审结合代码实现细节、接口设计、AI 协作日志评估自主设计能力

### 3.4 日志持久化强制规范

Agent 关键对话、工具调用、交互记录必须落地存储，采用**双目录**分工（与 `AGENTS.md` 一致）：

| 目录 | 写入方 | 内容 |
|------|--------|------|
| `.ai_history/sessions/` | omnidev-agent 运行时 | TUI/headless 会话快照 |
| `.ai_history/logs/` | 外部开发 AI 助手 | 协作开发 omnidev-agent 项目时的全量 JSONL/MD 日志 |

`.ai_history/logs/` 在开发阶段须随协作持续追加；运行时程序**不得**将会话写入 `logs/`（避免与协作日志混淆）。

## 6. 项目理解硬约束 (v2.0)

### 6.1 原则

修改既有项目文件前，Agent 必须先自动完成项目理解流程，将分析结果注入会话上下文。理解未完成前，// 调用被拦截。

### 6.2 检测方法

| 项目类型 | 检测条件 |
|----------|----------|
| Greenfield | 无构建文件、无  |
| Legacy | 存在构建文件 + 源码文件≥3 |

### 6.3 理解流程

自动执行四步：list_dir(/) → read_file(README) → search_code(关键模式) → read_file(入口文件)

## 7. SubAgent 并行调度 (v2.0)

### 7.1 能力

复杂需求由 **LLM 任务规划器** 自主判断：简单请求输出 **单任务**，可拆分/可并行的工作输出 **多任务 DAG**。无依赖的子任务 spawn SubAgent 并行执行。
每个 SubAgent 是独立 goroutine，共享父 Agent 安全策略，有独立 session context。

### 7.2 并行控制

- 最大并行数: 4
- 单 SubAgent 超时: 120s
- 单 SubAgent 最大推理轮次: 10

## 5. 验收标准

1. 程序可本地一键编译启动，TUI 界面正常渲染、交互无崩溃
2. 完整实现 Agent 闭环推理，可自主完成多步骤代码开发任务
3. 工具分级权限逻辑生效，高危操作强制弹窗确认，拒绝流程正常流转
4. 兼容严格 OpenAI 网关模式，支持流式输出、超时、重试容错
5. 会话快照自动持久化至 `.ai_history/sessions/`，协作日志写入 `.ai_history/logs/`，配置分层读取生效
6. 核心 Agent 循环、工具、权限、会话逻辑均为自主实现，未依赖第三方 Agent 框架
7. 测试用例覆盖全部指定场景，`deliverables/` 目录存在完整运行验证截图
8. 所有交付产物完整，目录结构符合规范，可直接提交评审
